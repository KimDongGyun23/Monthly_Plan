## 월간 회고

### 한달을 돌아보며

> 부족했던 부분

두 번째 달이 되니, 스스로의 부족한 부분이 어디인지 조금 더 명확히 보였다. 특히 목표를 세울 때 조금 더 구체적인 기준이 필요하다는 점을 깨달았다. `이건 했지` 하며 단순히 체크하고 넘어가는 경우가 많았기 때문이다.

<br />

예를 들어, `테스트 작성 및 관리 방법 학습` 같은 모호한 목표보다는 `테스트 작성 방법을 학습하고, 실제 프로젝트에 도입해보기` 처럼 실행 가능한 형태로 구체화된 목표가 더 효과적일 것 같다.

<br />

> 시간 관리의 필요성

또한 부스트캠프를 진행하며, 전체적인 목표를 완전히 달성하는 것이 쉽지 않다는 점도 느꼈다. 평일에는 부스트캠프 내 학습량이 많아 주로 주말에 추가 학습을 진행하고 있지만, 주말조차 관련 학습으로 이어져 시간의 여유가 부족하다는 한계가 있었다.

<br />

앞으로는 이러한 제약 속에서도 학습을 더 효율적으로 계획하고 실행할 방법을 고민해볼 필요가 있을 것 같다.

<br />

### Fetch API

> Fetch API 를 공부하게 된 배경

그동안 Fetch API 를 깊게 다뤄본 적이 없었다. 생각해보면, 이전 프로젝트에서는 대부분 Axios 를 기본 선택지처럼 사용해왔다. 새로운 프로젝트를 시작할 때마다 기존 코드를 복사해 붙여 사용했고, 자연스럽게 Axios 가 익숙한 기본 도구로 자리 잡게 되었다.

<br />

그렇다 보니 `왜 Axios 를 사용하는가?`, `Fetch 와의 차이는 무엇인가?` 같은 근본적인 질문을 던진 적이 거의 없었다. 단지 편리하다는 이유로 습관적으로 사용하고 있었던 것이다. 이 계기를 통해 Axios 와 Fetch 의 차이를 직접 비교하며 학습하고, 단순히 사용법을 익히는 것을 넘어 흐름을 이해하고자 하는 목표를 세우게 되었다.

<br />

> 새롭게 알게 된 부분이 있다면?

Fetch API 만으로도 Axios 가 제공하는 기능 대부분을 구현할 수 있었다. 이전에는 단순히 익숙함 때문에 Axios 를 사용했지만, 이번 학습을 통해 추가적인 라이브러리 없이도 충분히 같은 수준의 기능을 구현할 수 있음을 알게 되었다. 그렇다면 번들 크기를 늘리면서까지 Axios 를 사용할 필요가 있을까 하는 의문이 생겼다.

<br />

특히 이전까지 Axios 의 큰 장점이라고 생각했던 인터셉터 기능 역시 Fetch 를 활용해 직접 구현할 수 있었다. 요청과 응답 사이에서 필요한 로직을 주입하는 방식으로, 프레임워크에 의존하지 않고도 유연하게 처리할 수 있음을 확인했다.

<br />

이전에는 요청 취소 기능에 대해 `그럴 일이 얼마나 있을까?` 라며 깊이 있게 공부하지 않았었다. 하지만 이번 학습을 통해 규모가 큰 프로젝트일수록 요청 취소가 중요한 역할을 한다는 점을 깨달았다. 불필요한 네트워크 트래픽을 줄이고, 동시에 사용자 경험도 향상시킬 수 있기 때문이다.

<br />

이 경험을 통해 단순히 `어떤 기술을 쓰는가` 보다 `기술이 왜 필요한지` 를 생각해보는 것의 중요성을 다시금 느꼈다.

<br />

> 부족하다고 느낀 부분은?

앞서 언급한 것처럼, 요청 취소 기능이 대규모 프로젝트에서 매우 유용할 것이라고 생각했다. 그러나 이번 학습에서는 그 유용성을 실제로 검증하거나 수치로 측정해보지 못했다.

<br />

예를 들어, 사용자가 다른 페이지로 이동했음에도 이전 요청이 백그라운드에서 계속 진행된다면, 이는 불필요한 네트워크 사용으로 이어진다. 한두 명이라면 큰 문제가 없겠지만, 수천 명의 사용자가 동시에 같은 상황을 겪는다면 서버 입장에서도 부담이 누적된다.

<br />

이러한 현상을 실제로 측정하고 분석하는 과정이 있었다면, 요청 취소의 필요성을 보다 명확히 입증할 수 있었을 것이다. 단순히 `중요하다` 는 인식에 그치지 않고, 실제 데이터나 성능 지표로 그 효과를 확인하는 학습으로 확장했더라면 더 깊이 이해할 수 있었을 것 같다.

<br />

앞으로는 기능 구현뿐만 아니라, 측정과 검증을 통해 개선의 근거를 확보하는 학습 방향을 가지고 나아가고자 한다.

<br />

> 앞으로 어떻게 나아갈 것인가?

앞으로는 Axios 대신 Fetch API 를 기본 도구로 활용할 계획이다. 이번 학습을 통해 Fetch 만으로도 충분히 다양한 기능을 구현할 수 있고, 프로젝트의 의존성을 줄이며 더 가벼운 구조를 유지할 수 있다는 점을 확인했다.

<br />

또한, 학습한 내용을 바탕으로 기본 Fetch 설정을 공통 모듈로 정리하여 프로젝트 전반에 적용할 생각이다. 이를 통해 요청/응답 로직을 일관성 있게 관리하고, 필요 시 인터셉터나 요청 취소 같은 기능도 직접 구현해보며 확장해가고자 한다.

<br />

### 상태관리 패턴

> 상태관리 패턴을 학습하게 된 배경

프론트엔드를 처음 학습할 때는 Redux 를 사용하며 상태관리를 익혔다. 하지만 그 당시에는 원리를 깊이 이해하기보다는 강의나 예제 코드를 따라 구현하는 데 급급했다. 작성한 코드가 왜 그렇게 동작하는지, 어떤 구조에서 효율적인지에 대한 고민은 부족했다.

<br />

이후 Zustand 가 등장하며 `간단하고 직관적이다` 라는 이유로 이를 사용하기 시작했다. 실제로 코드가 훨씬 간결해지고 사용하기 편리했지만, 왜 간단한지, 어떤 원리로 상태를 관리하고 있는지에 대해서는 깊이 생각해보지 않았다. 단지 `편리하니까 쓴다` 수준에 머물러 있었던 것이다.

<br />

이번에 상태관리 패턴을 다시 학습해보며, 각 라이브러리가 해결하고자 하는 문제와 내부 동작 원리를 이해하고 싶었다. 단순히 도구를 사용하는 데서 그치지 않고, 도구가 왜 필요한지, 어떤 상황에서 어떤 패턴이 적합한지를 판단할 수 있는 기준을 세우기 위해서다.

<br />

> 새롭게 알게 된 부분이 있다면?

상태관리 패턴이 발전해온 흐름을 하나씩 따라가며 학습했다. 그 출발점은 기본이 되는 Flux 패턴이었다. Flux 의 구조와 철학을 이해하고 나니, 그 위에서 발전한 Redux 의 개념과 동작 원리가 한층 명확히 다가왔다.

<br />

이 과정을 통해, Redux 가 복잡하게 느껴졌던 이유 또한 자연스럽게 이해할 수 있었다. 그리고 그 복잡함을 해소하기 위해 등장한 다양한 경량 상태관리 라이브러리들 (Zustand, Recoil 등) 이 어떤 문제를 해결하고자 만들어졌는지도 알게 되었다.

<br />

서로 세부 구현은 다르지만, 모두가 애플리케이션의 데이터 흐름을 예측 가능하고 일관되게 만들기 위한 공통된 목표를 가지고 있다는 점이 인상 깊었다. 이 흐름을 따라가며 각각의 철학과 원리를 비교해보는 과정은, 단순히 문법을 익히는 것보다 훨씬 유익했다.

<br />

> 부족하다고 느낀 부분은?

여러 상태관리 라이브러리를 학습했음에도 불구하고, 여전히 Zustand 를 주로 사용하게 되는 이유가 완전히 명확하지 않다. 간편하고 직관적이라는 장점은 알고 있지만, 생각해보면 다른 라이브러리들도 충분히 간편하게 사용할 수 있다는 점에서 의문이 남았다.

<br />

따라서 앞으로는 단순히 사용성이 아니라, 왜 많은 개발자, 특히 취업을 준비하는 학생들이 Zustand 를 선택하는지 그 근본적인 이유를 더 깊이 탐구해보고자 한다. 많이 사용된다는 사실 뒤에는 명확한 철학이나 활용 목적, 혹은 학습 난이도의 차이가 있을 것이라 생각한다.

<br />

이 부분을 이해한다면 앞으로 어떤 상황에서 어떤 상태관리 패턴을 선택해야 하는지도 더 분명해질 것이다.

<br />

> 앞으로 어떻게 나아갈 것인가?

리액트에서 상태 는 곧 렌더링의 흐름과 직결되는 핵심 요소다. 그동안 리렌더링이 발생하거나 발생하지 않는 지점을 확인할 때마다, 직관과 감에 의존해 코드를 수정하며 문제를 해결하려는 경우가 많았다.

<br />

앞으로는 단순히 결과를 보고 맞춰 나가기보다는, 렌더링이 일어나는 원리와 상태 변화의 흐름을 깊이 이해하는 방향으로 학습하고자 한다. 즉, 코드를 무작정 수정하기보다 `왜 이런 현상이 발생했을까?` 를 먼저 생각하고, 그 원인을 논리적으로 분석한 뒤 수정하는 습관을 기르는 것이다.

<br />

### 테스트 작성

> 테스트를 학습해보고자 했던 배경

부스트캠프를 진행하면서 테스트 코드의 중요성을 자주 접하게 되었다. 특히 백엔드에서는 테스트 작성이 자연스럽고 필수적인 과정처럼 여겨졌지만, 프론트엔드에서는 언제, 어떻게 테스트를 적용해야 하는지에 대한 감이 부족했다.

<br />

이전까지는 `테스트는 개발 막바지에 안정성을 확인하기 위한 절차` 정도로만 생각했지만, 학습을 거듭할수록 테스트가 코드의 신뢰성과 유지보수성을 높이는 핵심 도구임을 깨달았다.

<br />

그럼에도 프론트엔드 환경에서는 DOM, 이벤트, 비동기 로직 등 테스트해야 할 대상이 다양하기 때문에, 어디까지 테스트해야 하는지, 어떤 수준의 테스트가 적절한지 명확히 구분하기 어려웠다.

<br />

> 새롭게 알게 된 부분이 있다면?

처음에는 백엔드 로직을 테스트하듯, 단순한 유틸 함수나 가벼운 커스텀 훅 정도만 테스트할 수 있을 것이라 생각했다. 하지만 실제로 테스트 코드를 작성해보며, UI 역시 모킹을 통해 테스트가 가능하다는 사실을 알게 되었다.

<br />

브라우저 환경에서의 렌더링이나 사용자 상호작용을 실제로 재현할 수 있다는 점이 인상 깊었다. 그동안 단순히 `이건 어떻게 테스트하지?` 라는 생각으로만 넘겼던 부분들이, 구현 가능한 구체적인 형태로 다가왔다.

<br />

단순히 기술을 배우기보다, 프론트엔드 테스트의 범위와 가능성을 새롭게 인식할 수 있었다.

<br />

> 부족하다고 느낀 부분은?

단위 테스트, 통합 테스트, E2E 테스트의 개념은 익혔지만, 막상 실제 프로젝트에 도입하려 하니 생각보다 쉽지 않았다. 단위 테스트로 접근하려 해도, 코드가 서로 얽혀 있는 부분이 많아 결국 통합 테스트에 가까워지는 상황이 자주 발생했다.

<br />

이 경험을 통해 `이 방식이 과연 올바른 테스트 접근일까?` 라는 의문이 생겼다. 테스트에 대한 이해가 아직 부족해서 그런 것인지, 아니면 테스트를 고려하지 않은 코드 구조 때문인 것인지 스스로 고민해볼 필요를 느꼈다. 테스트의 개념적 이해뿐만 아니라, `테스트하기 좋은 구조란 무엇인가` 에 대한 설계 관점에서 고민이 필요하다고 느꼈다.

<br />

> 앞으로 어떻게 나아갈 것인가?

백엔드 개발을 진행하다 보면, 하나의 로직을 완성한 뒤 테스트를 작성하거나, 경우에 따라 작성 전에 테스트를 설계하는 경우도 있다. 그렇다면 프론트엔드에서도 동일한 접근이 가능할까?

<br />

지금까지 프론트엔드에서는 테스트를 깊이 다뤄본 적이 없었기에, 이를 실제로 어떤 방식으로 적용할 수 있을지 학습해보고자 한다.

<br />

### 자기소개서

> 자기소개서를 작성해보고자 했던 배경

최근 들어 취업 준비가 전혀 되어 있지 않다는 생각이 들었다. 단순히 기술을 배우고 프로젝트를 수행하는 것에 집중해왔지만, 정작 이를 어떻게 표현하고 전달할지에 대한 준비는 부족했다.

<br />

그래서 스스로를 점검하고 정리하기 위해 자기소개서를 직접 작성해보며 연습해보고자 했다.

<br />

> 새롭게 알게 된 부분이 있다면?

막상 자기소개서를 작성해보니, 정해진 형식이 없는 경우가 많다는 사실을 새삼 느꼈다. 형식이 주어지지 않다 보니, 어떤 내용을 중심으로 써야 할지 막막했다. `나에 대한 이야기를 써야 할까?`, `프로젝트 경험을 중심으로 풀어야 할까?` 라는 질문이 계속 따라붙었다.

<br />

특히 프로젝트 경험을 소개하는 부분에서 혼란스러웠다. 이건 포트폴리오에 포함되어야 하는 내용이 아닌가? 하는 생각이 들었기 때문이다. 이 과정을 통해 자기소개서와 포트폴리오는 목적과 초점이 다르다는 점을 조금씩 이해하게 되었고, 앞으로는 두 영역을 구분해 정리할 필요성을 느꼈다.

<br />

> 부족하다고 느낀 부분은?

우선은 형식보다는 글을 쓰는 연습 자체에 집중했다. 그러나 막상 자기소개서를 직접 작성해보니 생각보다 쉽지 않았다. 어떤 부분을 강조해야 할지, 나를 가장 잘 드러낼 수 있는 요소가 무엇인지 계속 고민하게 되었다.

<br />

쓰다 보면 내용이 산만해지고, 이야기의 흐름이 뒤죽박죽 섞이는 느낌이 들었다. 내 경험을 논리적으로 정리하고 핵심을 중심으로 구성하는 글쓰기 연습이 앞으로 보완해야 할 가장 큰 과제라고 느꼈다.

<br />

> 앞으로 어떻게 나아갈 것인가?

하나의 페르소나를 설정하고 그 안에 나를 대입해보려 한다. 그 페르소나를 통해 나의 경험과 생각을 투영하면서, 나라는 사람을 하나의 ‘이야기’로 구조화해볼 계획이다.

<br />

이 과정에서 떠오르는 키워드와 경험들을 정리하고, 이를 연결해 서사적인 흐름을 만드는 연습을 하고자 한다. 단순한 자기소개서가 아니라, 나만의 스토리라인을 가진 인물로서 나를 표현하는 방법을 시도해보려 한다. 다음 달에는 이번 시도를 바탕으로 다시 한번 작성해볼 계획이다.

<br />

### 다음 달을 바라보며

> 꾸준히 학습하라

지금까지는 습관처럼 학습을 이어온 부분들이 많았다. 이제는 그 과정을 조금씩 다듬고 체계화해야 할 시점이다. 새로운 지식을 단순히 추가하는 것이 아니라, 기존 지식과 교체하는 방식으로 사고하며 하나씩 기반을 단단히 쌓아가자. 그렇게 점진적으로 발전하며, 더 튼튼한 구조를 만들어가고자 한다.

<br />

> 하나의 개념씩

이전에는 여러 개념을 한꺼번에 배우려다 보니 학습이 얕게 머무는 경우가 많았다. 이번에는 하나의 개념을 깊이 파고드는 학습을 시도해봤는데, 그 과정이 매우 유익했다. 앞으로도 한 번에 한 주제를 정하고, 그 카테고리를 충분히 이해하고 체화할 만큼 깊게 학습하는 방식을 꾸준히 이어나가고 싶다.

<br />

> 취업 준비

취업 준비는 아직 부족함을 많이 느낀다. 코딩테스트 연습, 포트폴리오 작성 등 미뤄둔 과제들을 조금씩 꾸준히 이어가야 할 시점이다. 이제는 물러설 곳이 없다는 마음으로, 채용 공고를 주기적으로 확인하고 필요한 역량을 보완해 나가며 준비해나가야 할 것이다.

<br />

> 하나의 마인드맵을 그려보자

한 달 단위로 하나의 목표 중심 마인드맵을 만들어보면 좋을 것 같다. 중심 키워드를 바탕으로 달성하고자 하는 목표를 도출하고, 그 목표를 세분화하며 실천 단계를 시각적으로 정리한다면 보다 명확한 방향성을 가질 수 있을 것이다.
